# Packaging UBL Viewer for Windows & the Microsoft Store

This document covers turning the Tauri build into installable artifacts, with the
Microsoft Store (MSIX) as the primary target because it unlocks the headline
feature: Windows offering UBL Viewer when a user opens a UBL invoice, and the
Store surfacing it to users who don't have it yet (§7–§8 of `INSTRUCTIONS.md`).

> **Prerequisites** (build host, Windows 10/11):
> - [Rust stable](https://www.rust-lang.org/) (`rustup default stable`)
> - [Bun](https://bun.sh) ≥ 1.3
> - **WebView2** runtime (preinstalled on Windows 11)
> - **MSVC build tools** (Visual Studio Build Tools, "Desktop development with C++")
> - For MSIX: the **Windows 10/11 SDK** (provides `makeappx.exe`, `signtool.exe`)
> - A **Microsoft Partner Center** account for Store submission

Everything below is run on a Windows host. The frontend (`bun run build`,
`type-check`, `lint`, `test`) is cross-platform; only the Rust/Tauri/MSIX steps
need Windows.

---

## 1. Standard Tauri build (MSI + NSIS)

```sh
bun install
bun tauri build
```

This produces, under `src-tauri/target/release/bundle/`:

- `msi/UBL Viewer_<version>_x64_en-US.msi` (WiX)
- `nsis/UBL Viewer_<version>_x64-setup.exe` (NSIS)

Both register the `.xml` **file association** declared in
`src-tauri/tauri.conf.json` (`bundle.fileAssociations`) as a *handler* — adding
UBL Viewer to the Explorer "Open with" menu without seizing the default. Good
for direct downloads and CI smoke tests; the Store path below is what enables
Store-driven discovery.

## 2. MSIX for the Microsoft Store

Tauri does not emit MSIX directly, so we package the built binaries ourselves.

### 2a. Icon assets

The MSIX manifest references tile logos plus a dedicated **file icon**
(`Assets\invoice.png`, §7c). The source artwork is
`src-tauri/icons/invoice.svg`. Rasterise it to PNG (e.g. with ImageMagick or
`resvg`) at the sizes the Store expects, and place them in the package's
`Assets\` folder alongside the existing tile logos generated by `bun tauri icon`:

```sh
# Example with resvg; any SVG→PNG rasteriser works.
resvg --width 256 --height 256 src-tauri/icons/invoice.svg Assets/invoice.png
```

### 2b. Fill in the manifest

Copy `src-tauri/msix/AppxManifest.xml` next to the staged binaries and replace
the `Identity` placeholders with the values from
**Partner Center → Product → Product identity**:

| Placeholder | Source |
| --- | --- |
| `Name` | Package/Identity/Name |
| `Publisher` | Package/Identity/Publisher (the `CN=…` string) |
| `Version` | `MAJOR.MINOR.BUILD.0` — keep in sync with `tauri.conf.json` |

Do **not** invent these — a mismatch fails certification.

### 2c. Stage and pack

Lay out a package root containing the executable, the WebView2 resources Tauri
produced, `AppxManifest.xml`, and `Assets\`:

```
staging/
  ubl-viewer.exe
  AppxManifest.xml
  Assets/
    Square44x44Logo.png  Square150x150Logo.png  StoreLogo.png
    Square71x71Logo.png  Square310x310Logo.png  Wide310x150Logo.png
    invoice.png
```

Then:

```sh
makeappx pack /d staging /p "UBL Viewer_<version>_x64.msix" /overwrite
```

### 2d. Sign (local testing only)

Partner Center re-signs on submission, so a trusted cert isn't needed for the
uploaded package. For **local install testing**, sign with a self-signed cert
whose subject matches `Identity/Publisher`:

```sh
signtool sign /fd SHA256 /a /f test-cert.pfx /p <pw> "UBL Viewer_<version>_x64.msix"
```

Install locally with `Add-AppxPackage .\UBL*.msix` (after trusting the cert),
then verify in Explorer:

1. Right-click a UBL `.xml` → **Open with** lists *UBL Viewer*.
2. Double-clicking after setting it as handler opens and renders the invoice.
3. A second double-click focuses the existing window (single-instance) and loads
   the new file in the same instance.
4. A non-UBL `.xml` opens to the "not a UBL invoice" state, not a crash (§6c).

### 2e. Submit

Build the upload bundle (`.msixupload`) and submit via Partner Center: reserve
the name, complete the listing, set pricing/availability, upload, and pass
certification.

## 3. CI / CD

- **`.github/workflows/ci.yml`** — runs `type-check`, `lint`, `test`, `build`
  (cross-platform) plus the Rust `cargo test` on every push/PR.
- **`.github/workflows/release.yml`** — builds the native installers on GitHub's
  runners and attaches them to a Release:
  - Publish a GitHub Release tagged `vX.Y.Z` → it builds **Windows** (`.msi` +
    `.exe`), **Linux** (`.deb` + `.AppImage`), and **macOS** (`.dmg`) and uploads
    them as release assets.
  - Or trigger it manually from the **Actions** tab (`workflow_dispatch`) to get
    the installers as downloadable workflow artifacts without cutting a release.

  Keep the release tag in sync with `version` in `tauri.conf.json` so the bundle
  filenames match. The Windows installers register the `.xml` *handler* via the
  NSIS/MSI association; the **MSIX** steps (2a–2e above) are the Store path and
  are not yet automated — add `makeappx`/`signtool` to a `windows-latest` job
  once Partner Center identity and signing secrets are configured.

> Evaluate [`cargo-packager`](https://github.com/crabnebula-dev/cargo-packager),
> which can emit MSIX and may streamline steps 2a–2c.
